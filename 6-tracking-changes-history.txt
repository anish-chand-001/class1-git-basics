61. USING git log
================================================================================

WHAT IS git log?
--------------------------------------------------------------------------------

The command used to view commit history.

Command:

git log

--------------------------------------------------------------------------------
EXAMPLE HISTORY
--------------------------------------------------------------------------------

Repository Contains:

Commit A

Commit B

Commit C

Commit D

Git Shows:

D

C

B

A

Newest commits appear first.

--------------------------------------------------------------------------------
EXAMPLE OUTPUT
--------------------------------------------------------------------------------

git log

commit 8d2f1a6c7a4b...

Author: Anish Chand

Date: Sun Jun 7 2026

    feat: add JWT authentication

------------------------------------------------

commit 3e5d7f1a...

Author: Anish Chand

Date: Sat Jun 6 2026

    fix: resolve login validation issue

--------------------------------------------------------------------------------
WHAT INFORMATION DOES git log SHOW?
--------------------------------------------------------------------------------

Every commit contains:

✓ Commit Hash
✓ Author
✓ Email
✓ Date
✓ Commit Message

--------------------------------------------------------------------------------
INDUSTRY USAGE
--------------------------------------------------------------------------------

Used daily for:

✓ Understanding Project History
✓ Finding Regressions
✓ Reviewing Previous Work
✓ Debugging Issues

================================================================================
63. UNDERSTANDING COMMIT HASHES
================================================================================

Every commit receives a unique identifier.

--------------------------------------------------------------------------------
EXAMPLE
--------------------------------------------------------------------------------

8d2f1a6f8f23a7bc9d2f4a1...

This identifier is called:

Commit Hash

--------------------------------------------------------------------------------
WHERE DOES IT COME FROM?
--------------------------------------------------------------------------------

Generated using:

SHA-1 Hashing

Based on:

✓ Snapshot
✓ Author
✓ Date
✓ Parent Commit
✓ Commit Message

--------------------------------------------------------------------------------
WHY HASHES MATTER
--------------------------------------------------------------------------------

Git identifies commits using hashes.

Example:

git show 8d2f1a6

Git knows exactly which commit you mean.

--------------------------------------------------------------------------------
INDUSTRY TIP
--------------------------------------------------------------------------------

Usually only first 7-10 characters are used.

Example:

8d2f1a6

instead of:

8d2f1a6f8f23a7bc9d2b6c4e8...

================================================================================
64. NAVIGATING COMMIT HISTORY
================================================================================

Git history forms a chain.

--------------------------------------------------------------------------------
EXAMPLE
--------------------------------------------------------------------------------

A ---> B ---> C ---> D ---> E

Current Position:

HEAD ---> E

--------------------------------------------------------------------------------
MOVING TO OLDER COMMITS
--------------------------------------------------------------------------------

Command:

git checkout 8d2f1a6

or

git switch --detach 8d2f1a6

--------------------------------------------------------------------------------
RESULT
--------------------------------------------------------------------------------

HEAD now points to:

Commit 8d2f1a6

instead of latest commit.

--------------------------------------------------------------------------------
INDUSTRY USAGE
--------------------------------------------------------------------------------

Useful for:

✓ Reproducing Bugs
✓ Investigating Old Code
✓ Testing Previous Releases
✓ Comparing Historical Behavior

================================================================================
67. USING git diff
================================================================================

One of the most important Git commands.

--------------------------------------------------------------------------------
PURPOSE
--------------------------------------------------------------------------------

Shows differences between versions.

--------------------------------------------------------------------------------
EXAMPLE
--------------------------------------------------------------------------------

Before:

const age = 20;

----------------------------------------

After:

const age = 21;

--------------------------------------------------------------------------------
COMMAND
--------------------------------------------------------------------------------

git diff

--------------------------------------------------------------------------------
OUTPUT
--------------------------------------------------------------------------------

- const age = 20;
+ const age = 21;

--------------------------------------------------------------------------------
MEANING
--------------------------------------------------------------------------------

- = Removed Line

+ = Added Line

--------------------------------------------------------------------------------
WHAT DOES git diff SHOW?
--------------------------------------------------------------------------------

By default:

Working Directory

VS

Staging Area

--------------------------------------------------------------------------------
INDUSTRY USAGE
--------------------------------------------------------------------------------

Before committing:

git diff

to review all changes.

================================================================================
68. COMPARING STAGED CHANGES
================================================================================

Normal diff:

git diff

shows:

Working Directory
vs
Staging Area

--------------------------------------------------------------------------------
VIEW STAGED CHANGES
--------------------------------------------------------------------------------

Command:

git diff --staged

OR

git diff --cached

--------------------------------------------------------------------------------
WHAT DOES IT SHOW?
--------------------------------------------------------------------------------

Staging Area

VS

Last Commit

--------------------------------------------------------------------------------
WORKFLOW
--------------------------------------------------------------------------------

Step 1:

git add .

----------------------------------------

Step 2:

git diff --staged

----------------------------------------

Step 3:

git commit

--------------------------------------------------------------------------------
WHY THIS MATTERS
--------------------------------------------------------------------------------

Verify exactly what will be committed.

Professional developers do this frequently.

================================================================================
69. COMPARING COMMITS
================================================================================

Compare two project snapshots.

--------------------------------------------------------------------------------
COMMAND
--------------------------------------------------------------------------------

git diff COMMIT1 COMMIT2

--------------------------------------------------------------------------------
EXAMPLE
--------------------------------------------------------------------------------

git diff a1b2c3 d4e5f6

--------------------------------------------------------------------------------
WHAT GIT SHOWS
--------------------------------------------------------------------------------

Everything that changed between:

Commit A

and

Commit B

--------------------------------------------------------------------------------
VISUAL
--------------------------------------------------------------------------------

Commit A
      |
      |
      v
  Changes
      |
      |
      v
Commit B

--------------------------------------------------------------------------------
INDUSTRY USAGE
--------------------------------------------------------------------------------

Very common during:

✓ Bug Investigations
✓ Release Reviews
✓ Security Audits
✓ Feature Analysis

================================================================================
70. COMPARING BRANCHES
================================================================================

Compare development work across branches.

--------------------------------------------------------------------------------
EXAMPLE BRANCHES
--------------------------------------------------------------------------------

main

feature-auth

--------------------------------------------------------------------------------
COMMAND
--------------------------------------------------------------------------------

git diff main feature-auth

--------------------------------------------------------------------------------
WHAT DOES IT SHOW?
--------------------------------------------------------------------------------

Changes present in:

feature-auth

compared to:

main

--------------------------------------------------------------------------------
VISUAL
--------------------------------------------------------------------------------

main
 |
 A ---> B ---> C

feature-auth
 |
 A ---> B ---> C ---> D ---> E

git diff main feature-auth

Shows:

D
E

Changes only.

--------------------------------------------------------------------------------
INDUSTRY USAGE
--------------------------------------------------------------------------------

Before merging:

Review branch differences.

Questions answered:

✓ What changed?
✓ How many files changed?
✓ What code was added?
✓ What code was removed?

--------------------------------------------------------------------------------
BEST PRACTICE
--------------------------------------------------------------------------------

Before opening a Pull Request:

Run:

git diff main feature-auth

Review everything carefully.

72. UNDERSTANDING SNAPSHOTS
================================================================================

This concept appears again because it is critical.

--------------------------------------------------------------------------------
COMMON MISCONCEPTION
--------------------------------------------------------------------------------

Many beginners think:

Git stores changes.

WRONG.

Git stores:

SNAPSHOTS

--------------------------------------------------------------------------------
EXAMPLE
--------------------------------------------------------------------------------

Commit A

app.js

auth.js

----------------------------------------

Commit B

app.js

auth.js

payment.js

--------------------------------------------------------------------------------
WHAT GIT STORES
--------------------------------------------------------------------------------

Snapshot A

Snapshot B

NOT:

"+ payment.js"

--------------------------------------------------------------------------------
MENTAL MODEL
--------------------------------------------------------------------------------

Every Commit = Photograph

Think:

Day 1

📸 Project Photo #1

----------------------------------------

Day 2

📸 Project Photo #2

----------------------------------------

Day 3

📸 Project Photo #3

Each commit is a complete project state.

================================================================================
73. UNDERSTANDING COMMIT HISTORY
================================================================================

Commit history is simply:

Connected Snapshots

--------------------------------------------------------------------------------
EXAMPLE
--------------------------------------------------------------------------------

A ---> B ---> C ---> D

--------------------------------------------------------------------------------
WHAT DOES EACH COMMIT KNOW?
--------------------------------------------------------------------------------

Each commit stores:

My Parent Commit

Example:

D knows C

C knows B

B knows A

--------------------------------------------------------------------------------
HOW git log WORKS
--------------------------------------------------------------------------------

HEAD
 |
 v
 D
 |
 v
 C
 |
 v
 B
 |
 v
 A

Git follows parent references backwards.

74. FILTERING LOGS
================================================================================

Large projects may contain:

100,000+

commits.

You need filtering.

--------------------------------------------------------------------------------
LAST 5 COMMITS
--------------------------------------------------------------------------------

git log -5

--------------------------------------------------------------------------------
ONE-LINE FORMAT
--------------------------------------------------------------------------------

git log --oneline

--------------------------------------------------------------------------------
SPECIFIC AUTHOR
--------------------------------------------------------------------------------

git log --author="Anish"

--------------------------------------------------------------------------------
SPECIFIC DATE RANGE
--------------------------------------------------------------------------------

git log --since="1 week ago"

--------------------------------------------------------------------------------
MORE EXAMPLES
--------------------------------------------------------------------------------

Last Month:

git log --since="1 month ago"

----------------------------------------

Yesterday:

git log --since="yesterday"

----------------------------------------

Specific Date:

git log --since="2026-06-01"

--------------------------------------------------------------------------------
INDUSTRY USAGE
--------------------------------------------------------------------------------

Questions like:

✓ Show me commits from last week
✓ Show me commits by John
✓ Show me recent changes

can be answered instantly.

--------------------------------------------------------------------------------

--------------------------------------------------------------------------------
USE CASE
--------------------------------------------------------------------------------

Trying to locate:

Authentication Feature

without manually reading hundreds of commits.

--------------------------------------------------------------------------------
OTHER EXAMPLES
--------------------------------------------------------------------------------

git log --grep="payment"

git log --grep="login"

git log --grep="security"

--------------------------------------------------------------------------------
INDUSTRY USAGE
--------------------------------------------------------------------------------

Very useful when:

✓ Locating Features
✓ Finding Bug Fixes
✓ Investigating Historical Changes

================================================================================
76. BLAME ANALYSIS (git blame)
================================================================================

One of the most powerful Git commands.

--------------------------------------------------------------------------------
COMMAND
--------------------------------------------------------------------------------

git blame app.js

--------------------------------------------------------------------------------
EXAMPLE OUTPUT
--------------------------------------------------------------------------------

8d2f1a6 (Anish Chand) line 1

8d2f1a6 (Anish Chand) line 2

3e5d7f1 (John Doe) line 3

--------------------------------------------------------------------------------
WHAT DOES IT SHOW?
--------------------------------------------------------------------------------

For every line:

✓ Commit Hash
✓ Author
✓ Date
✓ Line Content

--------------------------------------------------------------------------------
QUESTION IT ANSWERS
--------------------------------------------------------------------------------

Who changed this line?

--------------------------------------------------------------------------------
INDUSTRY EXAMPLE
--------------------------------------------------------------------------------

Production Bug:

A suspicious line appears.

Run:

git blame userController.js

Immediately discover:

✓ Who wrote it
✓ Which commit added it
✓ When it happened

--------------------------------------------------------------------------------
IMPORTANT RULE
--------------------------------------------------------------------------------

Use git blame to understand history.

NOT to blame people.

BAD:

--------------------------------------------------------------------------------

Commit small logical units.

Authentication Feature

BAD:

Authentication

all in one commit

BEST PRACTICE #3
--------------------------------------------------------------------------------

Review changes before committing.

Command:

git diff

================================================================================

BEST PRACTICE #4
--------------------------------------------------------------------------------

Review staged changes.

Command:

git diff --staged

BEST PRACTICE #5
--------------------------------------------------------------------------------

Use Conventional Commits.

Examples:

feat:

fix:

docs:

refactor:

test:

chore:

================================================================================

BEST PRACTICE #6
--------------------------------------------------------------------------------

Never rewrite shared history carelessly.

(Advanced Git topic)

Reason:

Other developers may depend on that history.
