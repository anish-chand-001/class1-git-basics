================================================================================
31. WHAT IS A REPOSITORY?
================================================================================

BEGINNER DEFINITION
--------------------------------------------------------------------------------

A Repository (Repo) is a storage space where Git tracks and manages a project.

Think:

Project + Git History = Repository

--------------------------------------------------------------------------------
Example
--------------------------------------------------------------------------------

ecommerce-app/
|
|-- package.json
|-- server.js
|-- routes/
|-- controllers/
|
+-- .git/

This entire project becomes a Git Repository.

--------------------------------------------------------------------------------
REAL DEFINITION
--------------------------------------------------------------------------------

A Git Repository contains:

SOURCE CODE
-----------
server.js
app.js
routes/
controllers/

HISTORY
--------
Commit A
Commit B
Commit C

BRANCHES
--------
main
feature-auth
feature-payment

TAGS
----
v1.0
v2.0

CONFIGURATION
-------------
.git/config

--------------------------------------------------------------------------------
INDUSTRY PERSPECTIVE
--------------------------------------------------------------------------------

A Repository usually represents:

✓ One Application
  Example: Netflix Backend

✓ One Service
  Example: Payment Service

✓ One Frontend
  Example: React Dashboard

✓ One Microservice
  Example: Order Service

--------------------------------------------------------------------------------
COMMON MISCONCEPTION
--------------------------------------------------------------------------------

Many beginners think:

Repository = Files

WRONG

Repository =

Files
+
History
+
Branches
+
Tags
+
Metadata

================================================================================
32. CREATING A REPOSITORY (git init)
================================================================================

This is where Git begins.

Command:

git init

--------------------------------------------------------------------------------
BEFORE
--------------------------------------------------------------------------------

ecommerce-app/
|
|-- server.js
|
+-- package.json

--------------------------------------------------------------------------------
RUN
--------------------------------------------------------------------------------

git init

--------------------------------------------------------------------------------
AFTER
--------------------------------------------------------------------------------

ecommerce-app/
|
|-- .git/
|-- server.js
|
+-- package.json

--------------------------------------------------------------------------------
WHAT HAPPENS INTERNALLY?
--------------------------------------------------------------------------------

Git creates:

.git/

This folder contains:

✓ Commits
✓ Branches
✓ Tags
✓ References
✓ Configuration
✓ History

--------------------------------------------------------------------------------
IMPORTANT RULE
--------------------------------------------------------------------------------

Without .git

= Normal Folder

With .git

= Git Repository

--------------------------------------------------------------------------------
INDUSTRY USAGE
--------------------------------------------------------------------------------

Starting a new project:

mkdir ecommerce-api

cd ecommerce-api

git init

Usually one of the first commands developers execute.

--------------------------------------------------------------------------------
VERIFY REPOSITORY
--------------------------------------------------------------------------------

Command:

git status

Example Output:

On branch main

No commits yet

This confirms Git is active.

================================================================================
33. CLONING A REPOSITORY (git clone)
================================================================================

In companies, most of the time you DO NOT create repositories.

You CLONE existing repositories.

--------------------------------------------------------------------------------
WHAT IS CLONING?
--------------------------------------------------------------------------------

Copying a repository from a remote source.

Command:

git clone repository-url

--------------------------------------------------------------------------------
EXAMPLE
--------------------------------------------------------------------------------

git clone https://github.com/company/backend-api.git

--------------------------------------------------------------------------------
WHAT DOES GIT DOWNLOAD?
--------------------------------------------------------------------------------

✓ Source Code
✓ Commit History
✓ Branches
✓ Tags
✓ Configuration

Everything.

--------------------------------------------------------------------------------
AFTER CLONING
--------------------------------------------------------------------------------

backend-api/
|
|-- .git/
|-- src/
|-- package.json
|
+-- README.md

--------------------------------------------------------------------------------
INDUSTRY WORKFLOW
--------------------------------------------------------------------------------

Day 1 at Company:

Manager:

"Clone the project and run it locally."

Developer:

git clone repository-url

cd project

npm install

npm start

This is often the very first task.

================================================================================
34. LOCAL REPOSITORY STRUCTURE
================================================================================

Typical Repository:

ecommerce-api/
|
|-- .git/
|-- src/
|-- package.json
|-- .gitignore
|
+-- README.md

--------------------------------------------------------------------------------
SOURCE CODE
--------------------------------------------------------------------------------

src/

Contains application code.

--------------------------------------------------------------------------------
README
--------------------------------------------------------------------------------

README.md

Contains:

✓ Project Overview
✓ Installation Steps
✓ Usage Instructions
✓ API Documentation

--------------------------------------------------------------------------------
GITIGNORE
--------------------------------------------------------------------------------

.gitignore

Contains files Git should ignore.

--------------------------------------------------------------------------------
PACKAGE FILE
--------------------------------------------------------------------------------

package.json

Contains:

✓ Dependencies
✓ Scripts
✓ Project Metadata

--------------------------------------------------------------------------------
GIT DIRECTORY
--------------------------------------------------------------------------------

.git/

Contains:

✓ Commits
✓ Branches
✓ Tags
✓ References
✓ Configuration

--------------------------------------------------------------------------------
INDUSTRY TIP
--------------------------------------------------------------------------------

Never manually edit:

.git/

unless you fully understand Git Internals.

35. WORKING WITH EXISTING PROJECTS
================================================================================

This is how most real-world development happens.

--------------------------------------------------------------------------------
SCENARIO
--------------------------------------------------------------------------------

You join a company.

Project already exists.

You DO NOT run:

git init

Instead:

git clone

--------------------------------------------------------------------------------
TYPICAL FLOW
--------------------------------------------------------------------------------

git clone repository-url

cd project

npm install

npm run dev

--------------------------------------------------------------------------------
BEFORE WRITING CODE
--------------------------------------------------------------------------------

Understand:

✓ Project Structure
✓ Current Branch
✓ README
✓ Environment Setup

--------------------------------------------------------------------------------
CHECK PROJECT STATUS
--------------------------------------------------------------------------------

git status

--------------------------------------------------------------------------------
CHECK CURRENT BRANCH
--------------------------------------------------------------------------------

git branch

--------------------------------------------------------------------------------
CHECK REMOTES
--------------------------------------------------------------------------------

git remote -v

Professional developers do this before making changes.

================================================================================
36. REPOSITORY BEST PRACTICES
================================================================================

A repository should be easy to understand.

--------------------------------------------------------------------------------
GOOD STRUCTURE
--------------------------------------------------------------------------------

project/
|
|-- src/
|-- tests/
|-- docs/
|-- README.md
|-- .gitignore
|
+-- package.json

--------------------------------------------------------------------------------
BAD STRUCTURE
--------------------------------------------------------------------------------

project/
|
|-- abc.js
|-- test1.js
|-- new.js
|-- final.js
|
+-- latest.js

Confusing.

--------------------------------------------------------------------------------
BEST PRACTICE #1
--------------------------------------------------------------------------------

Keep root folder clean.

GOOD:

src/
config/
tests/

BAD:

100 files in root directory

--------------------------------------------------------------------------------
BEST PRACTICE #2
--------------------------------------------------------------------------------

Use meaningful names.

GOOD:

userController.js
paymentService.js

BAD:

file1.js
new.js

--------------------------------------------------------------------------------
BEST PRACTICE #3
--------------------------------------------------------------------------------

Always include:

README.md

--------------------------------------------------------------------------------
BEST PRACTICE #4
--------------------------------------------------------------------------------

Always include:

.gitignore

--------------------------------------------------------------------------------
BEST PRACTICE #5
--------------------------------------------------------------------------------

Commit frequently.

Small commits are easier to:

✓ Review
✓ Debug
✓ Revert

38. PUBLIC VS PRIVATE REPOSITORIES
================================================================================

One of GitHub's most important concepts.

--------------------------------------------------------------------------------
PUBLIC REPOSITORY
--------------------------------------------------------------------------------

Anyone can:

✓ View
✓ Clone
✓ Fork

the code.

Examples:

✓ Open Source Projects
✓ Portfolio Projects
✓ Community Projects

--------------------------------------------------------------------------------
ADVANTAGES
--------------------------------------------------------------------------------

✓ Portfolio Building
✓ Open Source Contributions
✓ Community Feedback
✓ Learning Opportunities

--------------------------------------------------------------------------------
DISADVANTAGES
--------------------------------------------------------------------------------

✗ Code is visible to everyone

--------------------------------------------------------------------------------
PRIVATE REPOSITORY
--------------------------------------------------------------------------------

Only authorized users can:

✓ View
✓ Clone
✓ Modify

--------------------------------------------------------------------------------
USED FOR
--------------------------------------------------------------------------------

✓ Banking Software
✓ Payment Systems
✓ Internal Products
✓ Enterprise Applications

--------------------------------------------------------------------------------
INDUSTRY REALITY
--------------------------------------------------------------------------------

Most company repositories:

PRIVATE

Most portfolio repositories:

PUBLIC

39. LOCAL VS REMOTE REPOSITORIES
================================================================================

This concept causes confusion for beginners.

--------------------------------------------------------------------------------
LOCAL REPOSITORY
--------------------------------------------------------------------------------

Exists on your machine.

Example:

C:\Projects\ecommerce-app

Contains:

✓ Code
✓ History
✓ Branches

--------------------------------------------------------------------------------
REMOTE REPOSITORY
--------------------------------------------------------------------------------

Exists on a server.

Examples:

✓ GitHub
✓ GitLab
✓ Bitbucket

--------------------------------------------------------------------------------
VISUAL
--------------------------------------------------------------------------------

Developer
    |
    v
Local Repository
    |
    | Sync
    |
    v
Remote Repository

--------------------------------------------------------------------------------
WHY REMOTE REPOSITORIES EXIST
--------------------------------------------------------------------------------

1. BACKUP

Laptop dies?

Remote survives.

----------------------------------------

2. COLLABORATION

Multiple developers share code.

----------------------------------------

3. CI/CD

Builds and deployments often start from remote repositories.
