SECTION 10: MERGING & COLLABORATION
(141-160)
=========

# GOAL

Learn how multiple developers combine work safely into a single codebase.

Merging is the foundation of team collaboration in Git.

================================================================================
WHY MERGING EXISTS
==================

Imagine a company with three developers.

Developer A:
feature/jwt-authentication

Developer B:
feature/payment-gateway

Developer C:
feature/email-service

All branches started from the same codebase.

Eventually the company wants:

```
JWT Authentication
+
Payment Gateway
+
Email Service
```

inside one application.

The process of combining branch histories is called:

```
MERGING
```

================================================================================
141. UNDERSTANDING MERGE
========================

## DEFINITION

A merge combines changes from one branch into another.

Example:

main
│
├── feature-auth
│
└── feature-payment

After development:

main
│
├── feature-auth (completed)
│
└── feature-payment (completed)

Now combine them into main.

---

## COMMAND

Switch to target branch:

git switch main

Merge feature branch:

git merge feature-auth

---

## MENTAL MODEL

Road A
+
Road B
=
One Highway

================================================================================
142. FAST-FORWARD MERGE
=======================

This is the simplest merge type.

---

## BEFORE

A ---> B ---> C
^
main
^
feature-auth

Feature branch receives new commits:

A ---> B ---> C ---> D ---> E
^
feature-auth

main remains at C.

---

## COMMAND

git merge feature-auth

---

## AFTER

A ---> B ---> C ---> D ---> E
^
main
^
feature-auth

---

## WHY "FAST-FORWARD"?

Git simply moves:

main ---> E

No merge commit is created.

---

## ADVANTAGES

✓ Clean History

✓ No Extra Commit

✓ Easy To Understand

================================================================================
143. THREE-WAY MERGE
====================

Most common merge in real projects.

---

## BEFORE

Starting Point:

A ---> B ---> C

main receives:

A ---> B ---> C ---> D
^
main

feature-auth receives:

A ---> B ---> C ---> E
^
feature-auth

---

## WHAT GIT DOES

Git analyzes:

Common Ancestor = C

Current Branch = D

Feature Branch = E

---

## AFTER

A ---> B ---> C ---> D
\      
\      M
\    /
E

M = Merge Commit

================================================================================
144. MERGE COMMITS
==================

## NORMAL COMMIT

Has one parent.

A ---> B ---> C

Parent of C = B

---

## MERGE COMMIT

Has two parents.

```
   D
  /
```

A-B-C

E

Merge ---> M

Parents:

Parent 1 = D

Parent 2 = E

---

## VIEW MERGE COMMIT

git show MERGE_HASH

================================================================================
145. MERGE CONFLICTS
====================

## WHAT IS A CONFLICT?

Git cannot determine which change should win.

---

## EXAMPLE

Original:

const tax = 10;

Developer A:

const tax = 15;

Developer B:

const tax = 20;

Git asks:

Which value should be kept?

---

## RESULT

Conflict Created

Git Stops

Developer Must Decide

================================================================================
146. WHY CONFLICTS OCCUR
========================

## COMMON CAUSES

1. Same File Edited

auth.js modified by multiple developers.

---

2. Same Line Edited

Developer A:

const PORT = 3000;

Developer B:

const PORT = 5000;

---

3. Deleted vs Modified

Developer A deletes file.

Developer B modifies file.

---

4. Renaming Conflicts

Developer A:

user.js -> users.js

Developer B:

modifies user.js

================================================================================
147. RESOLVING CONFLICTS
========================

## CONFLICT MARKERS

<<<<<<< HEAD
const tax = 15;
===============

const tax = 20;

> > > > > > > feature-auth

---

## MEANING

HEAD

Current Branch

=======

Separator

feature-auth

Incoming Branch

---

## RESOLUTION PROCESS

Step 1:

Edit code manually.

Step 2:

Remove conflict markers.

Step 3:

Save file.

Step 4:

git add file.js

Step 5:

git commit

Merge completed.

================================================================================
148. CONFLICT RESOLUTION STRATEGIES
===================================

## STRATEGY 1

Accept Current Change

Keep your version.

================================================================================

## STRATEGY 2

Accept Incoming Change

Keep teammate's version.

================================================================================

## STRATEGY 3

Combine Both

Example:

const tax = 15;
const shipping = 20;

================================================================================

## STRATEGY 4

Rewrite Entire Section

Often safest.

---

## INDUSTRY RULE

Never resolve conflicts blindly.

Understand the business logic first.

150. MERGE VS REBASE
====================

## MERGE

Preserves branch history.

Example:

A-B-C

D-E

M

Command:

git merge feature-auth

---

## ADVANTAGES

✓ Safe

✓ History Preserved

✓ Beginner Friendly

================================================================================

## REBASE

Rewrites history.

Before:

A-B-C

D-E

After:

A-B-C-D-E

Command:

git rebase main

---

## ADVANTAGES

✓ Linear History

✓ Cleaner Logs

✓ Professional Commit Graph

---

## INDUSTRY RULE

Private Branch:

Use Rebase

Shared Branch:

Prefer Merge

Never rebase public shared history.
