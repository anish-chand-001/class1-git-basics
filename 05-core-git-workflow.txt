================================================================================
41. WORKING DIRECTORY
================================================================================

WHAT IS A WORKING DIRECTORY?
--------------------------------------------------------------------------------

The Working Directory is your actual project folder where you:

✓ Create Files
✓ Edit Files
✓ Delete Files

Example:

ecommerce-api/
|
|-- server.js
|-- package.json
|-- routes/
|
+-- controllers/

This is your Working Directory.

--------------------------------------------------------------------------------
REAL DEVELOPER EXAMPLE
--------------------------------------------------------------------------------

You open VS Code and write:

const express = require("express");

You are modifying files in:

Working Directory

Git has NOT:

✗ Saved anything
✗ Committed anything
✗ Tracked changes yet

You are simply editing files.

--------------------------------------------------------------------------------
INDUSTRY EXAMPLE
--------------------------------------------------------------------------------

Task:

Add JWT Authentication

Files Modified:

authController.js

authMiddleware.js

userRoutes.js

All changes first happen in:

Working Directory

--------------------------------------------------------------------------------
IMPORTANT CONCEPT
--------------------------------------------------------------------------------

Working Directory is:

MUTABLE

Meaning:

Files can change at any time.

================================================================================
42. STAGING AREA (INDEX)
================================================================================

One of the most misunderstood Git concepts.

--------------------------------------------------------------------------------
WHAT IS STAGING AREA?
--------------------------------------------------------------------------------

The Staging Area is a temporary holding area.

Think:

Shopping Cart

before checkout.

--------------------------------------------------------------------------------
EXAMPLE
--------------------------------------------------------------------------------

You changed:

authController.js

userRoutes.js

README.md

But you only want:

authController.js

userRoutes.js

in your next commit.

You stage only those files.

--------------------------------------------------------------------------------
WHY DOES GIT USE STAGING?
--------------------------------------------------------------------------------

Many Version Control Systems skip this layer.

Git gives developers more control.

--------------------------------------------------------------------------------
WITHOUT STAGING
--------------------------------------------------------------------------------

Edit Files
    |
    v
Commit Everything

--------------------------------------------------------------------------------
WITH STAGING
--------------------------------------------------------------------------------

Edit Files
    |
    v
Choose Files
    |
    v
Commit Selected Files

--------------------------------------------------------------------------------
INDUSTRY EXAMPLE
--------------------------------------------------------------------------------

You worked on:

1. JWT Authentication

AND

2. README Improvements

These should become:

Two separate commits.

Staging makes this possible.

--------------------------------------------------------------------------------
MENTAL MODEL
--------------------------------------------------------------------------------

Working Directory
      |
      v
Uncommitted Changes

----------------------------------------

Staging Area
      |
      v
Selected Changes

----------------------------------------

Repository
      |
      v
Saved History

Stored inside:

--------------------------------------------------------------------------------
WHAT HAPPENS AFTER COMMIT?
--------------------------------------------------------------------------------

Command:

git commit -m "feat: add JWT authentication"

Git creates:

✓ Commit Object
✓ Tree Object
✓ Blob Objects

and stores them inside:

.git/objects

--------------------------------------------------------------------------------
INDUSTRY ANALOGY
--------------------------------------------------------------------------------

Working Directory
=
Draft

----------------------------------------

Staging Area
=
Ready For Approval

----------------------------------------

Repository
=
Official Record

44. FILE LIFECYCLE IN GIT
================================================================================

One of the most important interview topics.

Every file moves through states.

--------------------------------------------------------------------------------
FILE LIFECYCLE
--------------------------------------------------------------------------------

Untracked
    |
    v
Staged
    |
    v
Committed
    |
    v
Modified
    |
    v
Staged
    |
    v
Committed

Understanding this lifecycle explains most Git commands.

================================================================================
45. UNTRACKED FILES
================================================================================

DEFINITION
--------------------------------------------------------------------------------

Git does not know this file exists.

--------------------------------------------------------------------------------
EXAMPLE
--------------------------------------------------------------------------------

Created:

paymentService.js

Run:

git status

Output:

Untracked files:
paymentService.js

--------------------------------------------------------------------------------
WHY?
--------------------------------------------------------------------------------

Git has never seen this file before.

--------------------------------------------------------------------------------
INDUSTRY EXAMPLE
--------------------------------------------------------------------------------

New developer creates:

emailService.js

Git marks it as:

UNTRACKED

until staged.

================================================================================
46. TRACKED FILES
================================================================================

DEFINITION
--------------------------------------------------------------------------------

Files Git already knows about.

Examples:

server.js

package.json

Previously committed files.

--------------------------------------------------------------------------------
TRACKED MEANS
--------------------------------------------------------------------------------

Git watches these files.

Whenever they change:

Git detects it.

================================================================================
47. MODIFIED FILES
================================================================================

DEFINITION
--------------------------------------------------------------------------------

A tracked file that has been changed.

--------------------------------------------------------------------------------
EXAMPLE
--------------------------------------------------------------------------------

Before:

console.log("Hello");

After:

console.log("Hello World");

Git detects:

Modified

Status:

modified: app.js

--------------------------------------------------------------------------------
INDUSTRY EXAMPLE
--------------------------------------------------------------------------------

Bug Fix:

userController.js

Git shows:

Modified

================================================================================
48. STAGED FILES
================================================================================

DEFINITION
--------------------------------------------------------------------------------

Files selected for the next commit.

--------------------------------------------------------------------------------
COMMAND
--------------------------------------------------------------------------------

git add userController.js

--------------------------------------------------------------------------------
STATE CHANGE
--------------------------------------------------------------------------------

Modified
    |
    v
Staged

--------------------------------------------------------------------------------
STATUS OUTPUT
--------------------------------------------------------------------------------

Changes to be committed:

userController.js

--------------------------------------------------------------------------------
IMPORTANT
--------------------------------------------------------------------------------

Staged does NOT mean committed.

Many beginners confuse these.

COMMAND
--------------------------------------------------------------------------------

--------------------------------------------------------------------------------
RESULT
--------------------------------------------------------------------------------

Changes become:

Committed

Meaning:

Saved permanently in Git history.

--------------------------------------------------------------------------------
NOW GIT CAN
--------------------------------------------------------------------------------

✓ Restore It
✓ Compare It
✓ Merge It
✓ Share It

================================================================================
50. USING git status
================================================================================

Most frequently used Git command.

--------------------------------------------------------------------------------
COMMAND
--------------------------------------------------------------------------------

git status

--------------------------------------------------------------------------------
PURPOSE
--------------------------------------------------------------------------------

Shows:

✓ Current Branch
✓ Modified Files
✓ Staged Files
✓ Untracked Files

--------------------------------------------------------------------------------
EXAMPLE OUTPUT
--------------------------------------------------------------------------------

On branch main

Changes to be committed:
authController.js

Changes not staged:
userRoutes.js

Untracked files:
paymentService.js

--------------------------------------------------------------------------------
INDUSTRY RULE
--------------------------------------------------------------------------------

Before:

git add

git commit

git push

Always run:

git status

first.

Senior developers use it constantly.

================================================================================
54. STAGING ALL FILES
================================================================================

COMMAND
--------------------------------------------------------------------------------

git add .

Stages everything in current directory.

--------------------------------------------------------------------------------
ALTERNATIVE
--------------------------------------------------------------------------------

git add -A

Stages entire repository changes.

--------------------------------------------------------------------------------
DIFFERENCE
--------------------------------------------------------------------------------

git add .

Current Directory Changes

----------------------------------------

git add -A

Entire Repository Changes

--------------------------------------------------------------------------------
INDUSTRY ADVICE
--------------------------------------------------------------------------------

Before using:

git add .

Always run:

git status

58. CONVENTIONAL COMMITS

Industry Standard Commit Format

--------------------------------------------------------------------------------
STRUCTURE
--------------------------------------------------------------------------------

type: description

--------------------------------------------------------------------------------
EXAMPLES
--------------------------------------------------------------------------------

test: add unit tests for payment service

--------------------------------------------------------------------------------

test      → Testing

style     → Formatting

perf      → Performance Improvement

--------------------------------------------------------------------------------
BENEFITS
--------------------------------------------------------------------------------

✓ Easier Releases
✓ Automated Changelogs

================================================================================
59. COMMIT BEST PRACTICES
================================================================================

RULE #1
--------------------------------------------------------------------------------

One logical change per commit.

--------------------------------------------------------------------------------
RULE #2
--------------------------------------------------------------------------------

Keep commits small.

BAD:

500 files changed

GOOD:

5–20 related files

--------------------------------------------------------------------------------
RULE #3
--------------------------------------------------------------------------------

Write meaningful commit messages.

--------------------------------------------------------------------------------
RULE #4
--------------------------------------------------------------------------------

Commit often.

--------------------------------------------------------------------------------
RULE #5
--------------------------------------------------------------------------------

Never commit secrets.

BAD:

.env

API Keys

Passwords

--------------------------------------------------------------------------------
RULE #6
--------------------------------------------------------------------------------

Ensure project builds successfully before commit.

--------------------------------------------------------------------------------
RULE #7
--------------------------------------------------------------------------------

Run tests before commit.

60. DAILY DEVELOPMENT WORKFLOW
================================================================================

This is what professional developers do every day.

--------------------------------------------------------------------------------
STEP 1 : GET LATEST CODE
--------------------------------------------------------------------------------

git pull origin main

--------------------------------------------------------------------------------
STEP 2 : CREATE FEATURE BRANCH
--------------------------------------------------------------------------------

git switch -c feature/jwt-authentication

--------------------------------------------------------------------------------
STEP 3 : WRITE CODE
--------------------------------------------------------------------------------

Implement feature.

--------------------------------------------------------------------------------
STEP 4 : CHECK CHANGES
--------------------------------------------------------------------------------

git status

--------------------------------------------------------------------------------
STEP 5 : REVIEW CHANGES
--------------------------------------------------------------------------------

git diff

--------------------------------------------------------------------------------
STEP 6 : STAGE CHANGES
--------------------------------------------------------------------------------

git add .

OR

git add authController.js

--------------------------------------------------------------------------------
STEP 7 : VERIFY STAGING
--------------------------------------------------------------------------------

git status

--------------------------------------------------------------------------------
STEP 8 : COMMIT
--------------------------------------------------------------------------------

git commit -m "feat: implement JWT authentication"

--------------------------------------------------------------------------------
STEP 9 : PUSH
--------------------------------------------------------------------------------

git push origin feature/jwt-authentication

--------------------------------------------------------------------------------
STEP 10 : CREATE PULL REQUEST
--------------------------------------------------------------------------------

Open PR for review.
