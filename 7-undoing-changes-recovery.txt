================================================================================
81. UNDERSTANDING HEAD
================================================================================

One of the most important Git concepts.

--------------------------------------------------------------------------------
WHAT IS HEAD?
--------------------------------------------------------------------------------

HEAD = Your Current Position

--------------------------------------------------------------------------------
EXAMPLE
--------------------------------------------------------------------------------

A ---> B ---> C ---> D
                   ^
                 HEAD

Currently you are at:

Commit D

--------------------------------------------------------------------------------
MORE ACCURATELY
--------------------------------------------------------------------------------

HEAD
 |
 v
main
 |
 v
 D

HEAD points to:

Branch

Branch points to:

Commit

--------------------------------------------------------------------------------
WHY HEAD MATTERS
--------------------------------------------------------------------------------

Almost every recovery command uses HEAD.

Examples:

git reset HEAD~1

git restore .

git checkout HEAD

git revert HEAD

--------------------------------------------------------------------------------
UNDERSTANDING HEAD~N
--------------------------------------------------------------------------------

HEAD~1

Means:

One Commit Before HEAD

Example:

A ---> B ---> C ---> D
                   ^
                 HEAD

HEAD~1 = C

HEAD~2 = B

HEAD~3 = A

================================================================================
82. UNSTAGING FILES
================================================================================

SCENARIO
--------------------------------------------------------------------------------

You accidentally staged a file.

Command:

git add app.js

Now:

Working Directory
        |
        v
   Staging Area

But you don't want it staged.

--------------------------------------------------------------------------------
MODERN SOLUTION
--------------------------------------------------------------------------------

git restore --staged app.js

--------------------------------------------------------------------------------
RESULT
--------------------------------------------------------------------------------

Removed From:

Staging Area

Still Exists In:

Working Directory

--------------------------------------------------------------------------------
IMPORTANT
--------------------------------------------------------------------------------

Code is NOT lost.

Only staging is removed.

--------------------------------------------------------------------------------
INDUSTRY USAGE
--------------------------------------------------------------------------------

Very common.

Used daily by developers.

================================================================================

SCENARIO
--------------------------------------------------------------------------------

--------------------------------------------------------------------------------
COMMAND
--------------------------------------------------------------------------------

--------------------------------------------------------------------------------

       v
Replaced By
       |
       v
Last Committed Version

--------------------------------------------------------------------------------
WARNING
--------------------------------------------------------------------------------

Uncommitted changes are lost.

--------------------------------------------------------------------------------
INDUSTRY USAGE
--------------------------------------------------------------------------------

Useful when experiments fail.

--------------------------------------------------------------------------------

================================================================================
85. USING git restore
================================================================================

Introduced to simplify Git.

--------------------------------------------------------------------------------
OLD APPROACH
--------------------------------------------------------------------------------

git checkout app.js

Confusing because checkout had multiple meanings.

--------------------------------------------------------------------------------
MODERN APPROACH
--------------------------------------------------------------------------------

git restore app.js

Much clearer.

--------------------------------------------------------------------------------
COMMON USAGE
--------------------------------------------------------------------------------

Restore File:

git restore app.js

----------------------------------------

Unstage File:

git restore --staged app.js

----------------------------------------

Restore Everything:

git restore .

--------------------------------------------------------------------------------
INDUSTRY RECOMMENDATION
--------------------------------------------------------------------------------

Prefer:

git restore

over older checkout syntax.

           HEAD

----------------------------------------

Working Directory:
Unchanged

Need to recommit.

================================================================================
88. HARD RESET
================================================================================

The most dangerous reset type.

--------------------------------------------------------------------------------
COMMAND
--------------------------------------------------------------------------------

git reset --hard HEAD~1

--------------------------------------------------------------------------------
BEFORE
--------------------------------------------------------------------------------

A ---> B ---> C

--------------------------------------------------------------------------------
AFTER
--------------------------------------------------------------------------------

A ---> B

Commit C disappears.

--------------------------------------------------------------------------------
WHAT HAPPENS?
--------------------------------------------------------------------------------

✓ Commit Removed

✓ Staging Removed

✓ Working Directory Reset

--------------------------------------------------------------------------------
DANGER
--------------------------------------------------------------------------------

Can destroy uncommitted work.

--------------------------------------------------------------------------------
INDUSTRY RULE
--------------------------------------------------------------------------------

Never run:

git reset --hard

unless you fully understand why.

================================================================================
89. RESET VS REVERT
================================================================================

Very common interview question.

--------------------------------------------------------------------------------
RESET
--------------------------------------------------------------------------------

Changes History

Example:

A ---> B ---> C

After Reset:

A ---> B

Commit C disappears.

--------------------------------------------------------------------------------
REVERT
--------------------------------------------------------------------------------

Preserves History

Example:

A ---> B ---> C
               |
               v
               D

D reverses C.

--------------------------------------------------------------------------------
INDUSTRY RULE
--------------------------------------------------------------------------------

Private Branch:

Use Reset

----------------------------------------

Shared Branch:

Use Revert

================================================================================
90. REVERTING COMMITS
================================================================================

COMMAND
--------------------------------------------------------------------------------

git revert COMMIT_HASH

Example:

git revert a1b2c3

--------------------------------------------------------------------------------
RESULT
--------------------------------------------------------------------------------

Before:

A ---> B ---> C

----------------------------------------

After:

A ---> B ---> C ---> D

D reverses changes from C.

--------------------------------------------------------------------------------
ADVANTAGE
--------------------------------------------------------------------------------

History remains intact.

--------------------------------------------------------------------------------
INDUSTRY USAGE
--------------------------------------------------------------------------------

Production bug fixes.

100. RECOVERY BEST PRACTICES
================================================================================

RULE #1
--------------------------------------------------------------------------------

Never panic.

Most Git problems are recoverable.

================================================================================

RULE #2
--------------------------------------------------------------------------------

Run:

git status

before doing anything.

================================================================================

RULE #3
--------------------------------------------------------------------------------

Use:

git diff

to understand changes.

================================================================================

RULE #4
--------------------------------------------------------------------------------

Learn:

git reflog

deeply.

It is the most powerful recovery tool in Git.

================================================================================

RULE #5
--------------------------------------------------------------------------------

Avoid:

git reset --hard

unless absolutely necessary.

================================================================================

RULE #6
--------------------------------------------------------------------------------

Prefer:

git revert

on shared branches.

================================================================================

RULE #7
--------------------------------------------------------------------------------

Create backup branches before risky operations.

Example:

git branch backup-before-reset

================================================================================

RULE #8
--------------------------------------------------------------------------------

Commit frequently.

Small commits are easier to recover.

================================================================================

When something goes wrong, ask:

--------------------------------------------------------------------------------
QUESTION 1
--------------------------------------------------------------------------------

Did I lose uncommitted work?

Recovery is harder.

--------------------------------------------------------------------------------
QUESTION 2
--------------------------------------------------------------------------------

Did I lose committed work?

Recovery is usually easy.

--------------------------------------------------------------------------------
WHY?
--------------------------------------------------------------------------------

Git remembers committed work through:

✓ Commit Graph

✓ References

✓ Reflog

THE ONE COMMAND TO REMEMBER
================================================================================

If you remember only one command from this section:

git reflog

--------------------------------------------------------------------------------
WHY?
--------------------------------------------------------------------------------

Many developers think Git is magic because it can recover
"deleted" work.

Reality:

Git rarely deletes committed data immediately.

MENTAL MODEL
--------------------------------------------------------------------------------

Just like an airplane flight recorder tracks where the plane went,

Reflog tracks where HEAD has been.

When disaster happens:

Check the reflog.
